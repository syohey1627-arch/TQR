# TQR Security Checklist

TQRは勤怠データと従業員情報を扱うため、本番公開前にこのチェックリストを通す。

## Supabase Database

- [ ] すべての公開テーブルでRLSを有効にする
- [ ] `anon` ロールで直接読める勤怠データを作らない
- [ ] `authenticated` ロールでも会社IDで必ず分離する
- [ ] 管理者以外が従業員・端末・打刻ログを編集できない
- [ ] `company_id` のない業務データテーブルを作らない
- [ ] 外部キー制約を入れる
- [ ] よく使う検索条件にindexを入れる
- [ ] 監査ログを残す

## Authentication

- [ ] 管理者ログインをSupabase Authに置き換える
- [ ] 端末ログイン方式を決める
- [ ] 本番PASSに `0000` や `1234` を使わない
- [ ] PASSは平文保存しない
- [ ] 管理者のパスワード再設定手順を用意する
- [ ] 退職者・無効ユーザーを停止できる
- [ ] 端末を無効化できる

## Employee PASS Punch

- [ ] 従業員PASSをハッシュ化する
- [ ] PASS認証失敗を記録する
- [ ] 短時間の連続失敗に制限をかける
- [ ] 会社・拠点単位でPASS打刻を無効化できる
- [ ] 管理画面でPASS打刻の利用状況を確認できる

## QR Punch

- [ ] QRに社員番号そのものを入れない
- [ ] QR tokenはランダム値にする
- [ ] DBにはtokenのhashを保存する
- [ ] 紛失QRを失効できる
- [ ] QR再発行履歴を残す

## Device Security

- [ ] 端末登録済み端末だけ打刻できる
- [ ] 端末IDと端末secretを発行する
- [ ] 紛失端末を無効化できる
- [ ] 端末ごとの最終同期日時を表示する
- [ ] オフライン保存データを暗号化する方針を決める
- [ ] Androidキオスクモードの運用手順を作る

## Punch Records

- [ ] 二重打刻を検出する
- [ ] 二重打刻は保存しない、または `blocked_duplicate` として保存する方針を決める
- [ ] 手動修正は元データを消さず、修正履歴を残す
- [ ] CSV出力履歴を残す
- [ ] 誰がいつ修正したか記録する

## Web Hosting

- [ ] HTTPSで公開する
- [ ] GitHub Pagesに本番秘密情報を置かない
- [ ] Supabase Service Role Keyをフロントに置かない
- [ ] Anon Keyだけをブラウザに置く
- [ ] 本番環境とテスト環境を分ける

## Before Production

- [ ] Supabase RLSの動作テストをする
- [ ] 別会社のデータが見えないことを確認する
- [ ] 管理者以外がCSV出力できないことを確認する
- [ ] 端末無効化後に打刻できないことを確認する
- [ ] PASS連続失敗の挙動を確認する
- [ ] バックアップ方針を確認する
- [ ] 利用規約・プライバシーポリシーを用意する
